## 1. Problem Statement

Family issues (money, chores, misunderstandings) are discussed ad-hoc in WhatsApp/calls with nothing written down. Emotions make tracking agreements hard.

**Family Council AI** solves this by:
- Letting members submit "cases" with text, audio, image, or video proof
- Using AI to summarize each case (what happened, who's involved, what's wanted)
- Building a weekly meeting agenda from all open issues
- Sending meeting invites via email
- Recording decisions and resolutions as a permanent history

**Dual purpose:** real family tool + portfolio project demonstrating ML/AI, backend, and infra skills.

## 3. Functional Requirements (MVP)

### 3.1 Auth
- Email + password (MVP); Google SSO later
- JWT-based sessions

### 3.2 Issue Management
- Create issue: title, description, attachments (audio/image/video/PDF)
- Issue states: `OPEN` → `IN_MEETING` → `RESOLVED` → `ARCHIVED`
- Store metadata in PostgreSQL, raw media in object storage (MinIO local / S3 cloud)

### 3.3 AI Summarization Pipeline
- Audio/video → Speech-to-Text (STT via API)
- Text + transcript → LLM summary (Groq, Llama 3.3 70B)
- Extract: people involved, key dates, topics (money/chores/health/education), requested resolution
- Store summary + structured fields in DB
- Optional later: embeddings in vector store for semantic search

### 3.4 Weekly Family Council Workflow
- Scheduler reads all `OPEN` issues
- Groups by topic/participant into a meeting agenda
- Creates a `Meeting` record with date/time, participants, ordered agenda
- Sends email via AWS SES with agenda + meeting link
- Optional: `.ics` calendar invite

### 3.5 Meeting Outcome Capture
- Admin marks each issue: resolved or needs follow-up
- Enters decision/outcome note (AI-assisted later)
- Issue moves to `RESOLVED`

### 3.6 History & Search
- List of past meetings and resolved issues
- Keyword search over summaries and outcomes (semantic search later)

## 4. Architecture

### Components

**Clients**
- Web: React + TypeScript
- Mobile: React Native / Expo (later phase — design APIs to be reusable)

**Backend (Python + FastAPI)**
- API service: auth, REST endpoints
- Issue Service: CRUD, attachments
- Meeting Service: agenda generation
- Notification Service: emails, calendar invites

**ML / Processing (added progressively)**
- Transcription: STT API (Groq Whisper or similar)
- Summarization: Groq LLM (Llama 3.3 70B)
- Agenda Builder: groups issues by topic/participants

**Data Stores**
- PostgreSQL: users, issues, attachments, meetings, decisions
- MinIO (local) / S3 (cloud): raw media files
- Vector store: Qdrant (later, for semantic search)

**Eventing (later phase)**
- Kafka topics: `issue_created`, `transcription_completed`, `summary_generated`, `meeting_scheduled`

**Infra**
- Docker Compose (now)
- Kubernetes (later)
- CI: GitHub Actions
- Email: AWS SES
